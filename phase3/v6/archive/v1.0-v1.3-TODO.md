### CLI 设计

  Embedding coverage:     98.4%  (2,549 / 2,591)

### 6 个指标的数据源

| 指标 | SQL / 数据源 | 复杂度 |
|------|--------------|--------|
| `total_chunks` | `SELECT COUNT(*) FROM chunks` | 1 行 |
| `embedding_coverage` | `chunks.vec_index IS NOT NULL` 占比 | 2 行 |
| `hit_rate` | `usage_count > 0` 占比（30 天窗口：`created_at > now - 30d OR last_used_at > now - 30d`）| 1 行 |
| `avg_inject_token` | 读取 `~/.hermes/memory/hermem_inject_log.jsonl` 最近 7 天，avg(len//4) | 文件读取 + parse |
| `dedup_rate` | 读取 V5.5 disposition 提取 outcome 字段（新加），7 天窗口 | 需 V5.5 配合加 outcome 字段，**如未就绪则返回 null + 提示"outcome 字段待 V5.5 补充"** |
| `ollama_latency` | 调 `ollama.ps()` 或 `ollama.embeddings()` 测一次往返 | 网络调用 |

### 实现位置

`hermes-agent/plugins/memory/hermem/cli.py`：

```python
# ── memory stats ─────────────────────────────────────────────────────────────

def _cmd_stats(args) -> int:
    """Handle `hermes memory stats`."""
    _setup_path()
    try:
        from impl.database import get_db, get_chunk_count
        from impl.vectorstore import get_stats
        from impl.embedding import is_ollama_healthy
    except Exception as e:
        print(f"[Hermem] stats failed to load modules: {e}")
        return 1

    # 1. total_chunks
    total = get_chunk_count()

    # 2. embedding_coverage
    with get_db() as conn:
        mapped = conn.execute(
            "SELECT COUNT(*) FROM chunks WHERE vec_index IS NOT NULL"
        ).fetchone()[0]

    # 3. hit_rate (30d)
    with get_db() as conn:
        hit_30d = conn.execute(
            """SELECT COUNT(*) FROM chunks
               WHERE usage_count > 0
                 AND (last_used_at > datetime('now', '-30 days')
                      OR created_at > datetime('now', '-30 days'))"""
        ).fetchone()[0]

    # 4. avg_inject_token
    log_path = Path.home() / ".hermes" / "memory" / "hermem_inject_log.jsonl"
    avg_token = _compute_avg_inject_token(log_path, days=7)

    # 5. dedup_rate (7d) — V5.5 配合加 outcome 字段前返回 None
    dedup_rate = _compute_dedup_rate(days=7)

    # 6. ollama_latency
    try:
        health = is_ollama_healthy()
        ollama_latency = health.get("latency_ms")
    except Exception:
        ollama_latency = None

    metrics = {
        "total_chunks": total,
        "embedding_coverage": round(mapped / total, 4) if total > 0 else 0.0,
        "hit_rate_30d": round(hit_30d / total, 4) if total > 0 else 0.0,
        "avg_inject_token_7d": avg_token,
        "dedup_rate_7d": dedup_rate,
        "ollama_latency_ms": ollama_latency,
    }

    if args.json:
        print(json.dumps(metrics, ensure_ascii=False, indent=2))
    else:
        _print_stats_table(metrics, total, mapped, hit_30d)

    return 0


def register_cli(subparser) -> None:
    # ... 已有 feedback / memory health / memory rebuild ...
    # 新增：
    stats_sub = memory_parser.add_subparser(
        "stats",
        help="Show Hermem memory usage metrics (hit rate, token, dedup, etc.)",
        description="Outputs 6 minimal metrics for observability. Use --json for machine-readable output.",
    )
    stats_sub.add_argument("--json", action="store_true")
    stats_sub.set_defaults(func=lambda args: _cmd_stats(args))
```

```python
def _v5_inject_chunk(self, chunk: dict) -> None:
    """将检索到的 chunk 直接注入到 prefetch result。"""
    sim = chunk.get("similarity", 0.0)
    content = chunk['content']
    injection = (
        f"\n\n[自动回忆 - 相似度 {sim:.2f}]\n"
        f"以下是从历史记忆中检索到的相关内容（可能相关，仅供参考）：\n"
        f"- {content}\n"
    )
    with self._prefetch_lock:
        current = self._prefetch_result
        if current:
            self._prefetch_result = current + injection
        else:
            self._prefetch_result = injection

    # V6 Sprint 0：记录 inject token 估算（喂给 hermes memory stats）
    try:
        from pathlib import Path
        import json
        from datetime import datetime, timezone
        log_path = Path.home() / ".hermes" / "memory" / "hermem_inject_log.jsonl"
        log_path.parent.mkdir(parents=True, exist_ok=True)
        with log_path.open("a", encoding="utf-8") as f:
            f.write(json.dumps({
                "ts": datetime.now(timezone.utc).isoformat(),
                "chunk_id": chunk.get("chunk_id", ""),
                "session_id": chunk.get("session_id", ""),
                "similarity": sim,
                "token_est": len(content) // 4,  # 粗估
                "char_count": len(content),
            }, ensure_ascii=False) + "\n")
    except Exception as e:
        # 日志失败不阻断主流程
        logger.debug("[Hermem V6] inject log write failed: %s", e)

    logger.debug("[Hermem V5] injected: sim=%.2f, content=%s",
                sim, content[:40])
```

### v1.3 借鉴点 1：RRF 融合公式（**作为 Sprint 1.1 增强，不新增任务编号**）

| 项 | 详情 |
|---|---|
| 借鉴源 | Hindsight 论文 §4.2.3：RRF(f) = Σ 1/(k+rank_i(f))，k=60 |
| 目的 | 替换 Hermem 当前 `search_with_tier` 的"硬阈值切分"，实现真正的"双路融合" |
| 当前 | `search_with_tier` 用单一 vec 通道 + 分层阈值（HIGH=0.70 / MEDIUM=0.50）|
| 升级 | HNSW 召回的向量候选 ∩ BM25 召回的关键词候选 → 各自排序 → RRF 融合 → 单一分数 |
| 公式 | `rrf_score(d) = 1/(60+rank_vec(d)) + 1/(60+rank_bm25(d))`，未出现在任一通道的 doc 分数为 0 |
| 落地 | **Sprint 1 任务 1.1 增强**（不新增任务，直接替换现有召回逻辑）|
| 涉及文件 | `phase3/impl/vector_search.py` `search_with_tier()` |
| 验收 | 单元测试：同一 doc 出现在两路 vs 仅一路 → 分数差异 ≥ 2x；50 条 ground-truth 上 recall@5 对比旧版 |
| 风险 | RRF 公式简单，调试性差；阈值仍需调（k=60 是否合适需要 sweep）|
| 与 v1.1 MemPalace 关系 | **不冲突**：MemPalace 用 `0.6×vec + 0.4×bm25` 加权融合，RRF 是 rank-based；两者二选一。**建议 RRF**（不需要校准通道分数，Hindsight 论文论证 RRF 在通道分数不可校准时更稳）|

**为什么不是 MemPalace 那种加权融合**：
- 加权融合需要"通道分数可校准"前提，vec cosine ∈ [-1, 1]、BM25 ∈ [0, ∞) —— 单位、分布全不同，必须先归一化
- RRF 只用 rank，绕开校准问题
- Hindsight 论文 §4.2.3 给了 4 条理由（rank-based、robust to missing、no penalty、multi-evidence rise），论证扎实

### v1.3 借鉴点 2：Temporal 检索通道（**Sprint 1 任务 1.5 新增**）

| 项 | 详情 |
|---|---|
| 借鉴源 | Hindsight 论文 §4.2.2：用规则+flan-t5-small 把 "yesterday"/"last weekend"/"June 2024" 解析为 [τ_start, τ_end]，再过滤 fact 的 occurrence 区间 |
| 目的 | 补 Hermem 当前"时间只影响排序权重"—— 改为"时间能过滤召回集" |
| 当前 | `usage_tracker.py` 记录 `last_used_at` 但只做 recency 加权，不做时间区间硬过滤 |
| 升级 | 给 `hermem_search` 加可选 `time_range: tuple[datetime, datetime] \| None` 参数。SQLite 层 `chunks.created_at BETWEEN ?` 直接过滤 |
| 简化版（v1.3）| **不引入 flan-t5-small**（Hindsight 用它兜底规则解析不了的 hard case），Hermem 的查询多数是中文/英文混合时间词，规则 + Python `dateparser` 库（轻量、pip install dateparser）足够 |
| 落地 | **Sprint 1 任务 1.5**（新增）|
| 涉及文件 | `phase3/impl/database.py`（查 chunks 时加 BETWEEN 条件）+ `phase3/impl/retrieval.py`（hermem_search API 加 time_range 参数）+ `hermes-agent/plugins/memory/hermem/__init__.py`（桥接层暴露）|
| 验收 | 单元测试：time_range=(2026-01-01, 2026-03-31) → 只返回 created_at 在区间的 chunk；与 RRF 融合顺序：先 Temporal 过滤再 RRF 融合（不是融合后再过滤）|
| 风险 | dateparser 库约 200KB 依赖；如不愿加依赖可手写 5-6 个 regex（"上周"/"上个月"/"YYYY-MM"/"Q1 2026"）|
| 与 V6 已有方向关系 | 加速 V6 Sprint 1（按需触发）的"anchor 关键词"信号 —— "上次"/"昨天做的"等显式时间词可直接走 Temporal 通道，不必走 anchor 兜底 |

**为什么不抄 Hindsight 的 Graph 通道（spreading activation）**：
- Hindsight 在 PostgreSQL 建完整 entity 图（mention → canonical entity + 4 类边），Hermem 没有 canonical entity 表
- 建 entity 图要先做实体识别（spaCy / LLM），引入额外依赖
- 个人 memory 场景下，**单跳 BM25+vector 融合已经覆盖 80% 召回需求**，entity graph 是 long-tail 优化
- **明确不借鉴进 V6**；进 V7+ backlog

### v1.3 借鉴点 3：Reflect 模式按需 API（**Sprint 3 任务 3.5 新增，与可解释包装合并**）

| 项 | 详情 |
|---|---|
| 借鉴源 | Hindsight 论文 §5：Reflect(B, Q, Θ) → (r, O') —— 调一次 LLM，retrieve + 生成综合答案 + 顺便形成 opinion |
| 目的 | 给 Hermem 加一个"按需深度回忆"模式，类似人脑"主动回忆"（主动搜索记忆 vs 被动触发）|
| 当前 | Hermem 只有 `hermem_search`（返回 chunk），不主动生成答案 |
| 升级 | 新增 `hermem_reflect(query, context_window) -> str`，行为：先 4 路召回（vec+bm25+temporal+rrf）→ top-k 拼 context → 调一次 LLM（Ollama 本地）→ 返回综合答案。**可选**把生成的答案存为 L4 reflection |
| 与 CARA 区别 | **不抄 CARA 的 personality profile (S/L/E/β)** —— Hermem 是个人 memory 不是 character AI，4 维 personality 是 over-engineering。Hermem 用 user_profile.md（L3 画像）做行为引导 |
| 与现有路径关系 | 与 `hermem_search` 平行。`hermem_search` 仍返回 chunks（agent 自己加工），`hermem_reflect` 返回答案（agent 拿到即用）|
| 落地 | **Sprint 3 任务 3.5**（新增，与 V6 Sprint 3 的 Explainable Wrapper 合并实现）|
| 涉及文件 | `phase3/impl/reflect.py`（新）+ `hermes-agent/plugins/memory/hermem/__init__.py`（暴露）|
| 验收 | 单元测试：调 hermem_reflect("上周讨论的 X 进展如何") 返回答案包含 1+ 引用 chunk；LLM 调用超时/失败时降级到 `hermem_search` 返回 chunks |
| 风险 | LLM 调用成本（Ollama 1.5B 也至少 1-2 秒）；与 L4 reflection cron 重复（需明确边界：reflect 是"当下"、L4 reflection 是"批量") |

### v1.3 不借鉴清单（4 个 — 避免工程栈/哲学错位）

| 不借鉴 | 原因 |
|---|---|
| **2.1 四网络（World/Experience/Opinion/Observation）**| Hindsight 哲学是"事实 vs 信念"严格分离，配套 personality profile 才能用。Hermem 是个人 memory，没有"信念需要演化"的需求；L0-L4 分层已经做了"原始-聚合-确认-反思"分离，**不是同一抽象** |
| **2.2 CARA personality profile (Skepticism/Literalism/Empathy/β)**| Hindsight 给 AI 配"性格"用于"扮演不同 customer service persona"。Hermem 是单 agent 服务单用户，没有多 persona 需求；用 user_profile.md 引导行为更轻量 |
| **2.3 Opinion reinforcement 公式 26**（c 分数 + α 步长）| Hindsight 用"信念置信度随新证据强化/削弱"。Hermem 是事实库不是 belief 系统，usage_count 已经是"被用过几次"的代理指标，不需要置信度框架 |
| **2.4 LLM 抽 narrative full text 存每条 fact** | Hindsight 每条 fact 都让 LLM 重写成 narrative 段落。Hermem 当前 `l1_extract.py` 已经在做类似事，但**只在 retain 时跑一次** —— 与 Hindsight 一致。问题：Hindsight 论文没给成本数据，但每条 fact 多一次 LLM 调用叠加起来成本可观。Hermem 走 Ollama 本地 1.5B 不在乎 LLM 成本，但**存储膨胀是真实成本**（每条 fact 多 200-500 token narrative），不抄 |

**额外警示**（写在 SPEC 里给 V7+ 的人参考）：
- Hindsight 论文 + README 已经**改写了核心术语**（论文里的 Opinion 在 README 里叫 Mental Models）。任何引用 Hindsight 论文做决策时，**先看当前代码版本，再看论文**。Hermem 不应该有这种版本漂移
- Hindsight 的"evaluation"用 GPT-OSS-120B 作为 judge 评自家答案（论文 §7.2），baseline 数字直接抄对方报告。**Hermem 的评测框架（Sprint 4）必须独立复现、不抄数字**，避免重蹈"自评自 SOTA"
